## Commands

- `npm run dev` ‚Äî start Next.js dev server (port 3000)
- `npm run build` ‚Äî production build
- `npm run lint` ‚Äî ESLint
- `npm run test` ‚Äî run all tests (`tsx --test src/**/*.test.ts`)
- `npm run sync:wordlists` ‚Äî regenerate `src/lib/wordlists.ts` from source data files
- `npm run fetch:answers` ‚Äî fetch confirmed answers from NYT API, update `answersByDate.ts`
- `npm run fetch:allowed` ‚Äî fetch complete valid-guess list from NYT Wordle JS bundle, update `allowed_words.txt`
- `npm run benchmark` ‚Äî full benchmark: play all historical games (~2-3 hours)
- `npm run benchmark:quick` ‚Äî quick benchmark: skip lookahead (~20 min)

## Architecture

Next.js 16 app (App Router) that helps solve daily Wordle puzzles using an entropy-based solver with weighted priors. Two routes: solver (`/`) and answer history (`/history`).

### Core solver pipeline

1. **Feedback engine** (`src/lib/wordle/feedback.ts`) ‚Äî computes Wordle tile patterns (B/Y/G) for a guess+answer pair, handling duplicate-letter rules correctly. `Pattern` is a 5-char string of `B|Y|G`.

2. **Solver** (`src/lib/wordle/solver.ts`) ‚Äî entropy scoring and candidate filtering. `bestNextGuess` picks the guess maximizing information entropy over the candidate set.

3. **Fast solver** (`src/lib/wordle/solver_fast.ts`) ‚Äî production solver used by the web worker. Uses a two-phase approach: heuristic letter-frequency shortlisting (top N from ~13K allowed guesses), then full entropy scoring on the shortlist. Adds **two-step lookahead** when candidates <= 200 to optimize for 2-3 guess solves.

4. **Web Worker** (`src/lib/wordle/solverWorker.ts`) ‚Äî runs `bestNextGuessHeuristic` off the main thread. Communicates via typed messages (`WorkerRequest`/`WorkerResponse`). The main page caches the first guess in localStorage keyed by date.

5. **Worker protocol** (`src/lib/wordle/workerProtocol.ts`) ‚Äî shared utilities for stale-response detection, first-guess caching logic, and candidate-set fallback (canonical -> broad).

### Candidate weighting

Shared weight computation in `src/lib/wordle/weights.ts` ‚Äî used by both the production solver and benchmark engine. `WeightConfig` controls three multiplicative factors:
- **Past-answer weight** (`history.ts`) ‚Äî NYT reuses past answers at ~11% rate (since Feb 2026); past answers get reduced weight (0.04 for used-once, 0.01 for used-twice) via `pastAnswerWeight(useCount, config?)`
- **Word frequency** (`wordFrequency.ts`) ‚Äî bigram/unigram scoring as a commonality proxy, maps to [0.2, 1.0]
- **Seasonal boosts** (`seasonalBoost.ts`) ‚Äî small multiplier for thematically relevant words by month/date

Production default (`DEFAULT_WEIGHT_CONFIG`): past-answer weight only ‚Äî benchmark showed frequency and seasonal priors hurt performance.

### Wordlists

`src/lib/wordlists.ts` is **auto-generated** by `scripts/sync-wordlists.mjs`. Do not edit it directly.
- `POSSIBLE_WORDS` (~2345) ‚Äî likely Wordle answers
- `ALLOWED_WORDS` (~14.9K) ‚Äî all valid guesses including probe words, fetched from NYT's Wordle JS bundle
- `NON_CANONICAL_ANSWERS` ‚Äî set of words NYT used as answers that weren't in the original curated pool (e.g., MOOCH, OASIS). Sourced from `answer_overrides.txt`.
- Source data: `src/data/allowed_words.txt`, `src/data/answers_by_date_wordlist.txt`
- To add missing NYT answers, add to `src/data/answer_overrides.txt` then run `npm run sync:wordlists`
- To refresh the full valid-guess list: `npm run fetch:allowed` then `npm run sync:wordlists`
- A biweekly GitHub Action (`.github/workflows/update-allowed-words.yml`) runs `fetch:allowed` + `sync:wordlists` on the 1st and 15th of each month, with automatic rollback on failure.

**Sync chain**: When updating word data, all downstream artifacts are regenerated by `sync:wordlists`. The Explore tab (`topGuesses` in `solver.ts`) uses heuristic shortlisting to score the full ALLOWED_WORDS pool, so no manual truncation is needed.

### Answer history

`src/lib/wordle/answersByDate.ts` is **auto-generated** by `scripts/fetch-answers.mjs`. Do not edit it directly. Contains the ordered list of all confirmed past Wordle answers (index 0 = 2021-06-19, "cigar"), fetched from the NYT API. `history.ts` uses this to compute past-answer use counts and weights based on a 5:00 AM UTC rollover.

**Data integrity**: The answer list MUST be API-verified. The original Wordle source code contained a pre-determined answer list that diverges from what NYT actually uses (NYT swaps planned words and reuses old ones). To re-verify the full list, clear `src/data/answers_by_date_wordlist.txt` then run `npm run fetch:answers`. The list may contain duplicate entries ‚Äî this is correct, as NYT reuses past answers (~11% rate since Feb 2026).

A daily GitHub Action (`.github/workflows/update-answers.yml`) runs `fetch:answers` + `sync:wordlists` at 5:30 AM UTC and auto-commits new answers. The fetch script is idempotent ‚Äî it reuses existing entries and only fetches dates not yet in the local list.

### Sentences

`src/data/sentences.json` contains one sentence per word in POSSIBLE_WORDS (~2,345 entries). Shown as a reward after solving. Each entry has `word`, `style`, and `sentence`. Styles rotate across 6 buckets (dry_wit, deadpan_dark, gallows_humor, sardonic, pop_reference, fun_fact) assigned alphabetically via `index % 6`. Lookup module at `src/lib/wordle/sentences.ts`. Generated by `scripts/generate-sentences.mjs` using the claude CLI. Spec at `wordle-sentence-spec.md`.

### UI

Tailwind CSS v4 for styling.

**Solver** (`src/app/page.tsx`) ‚Äî client component. Features: tile-tap feedback entry, undo/reset, guess history with colored tiles, visual keyboard, top-N guess explorer, confidence indicator, analysis drawer. Links to history page (opens in new tab to preserve solver state).

**Answer History** (`src/app/history/page.tsx`) ‚Äî client component. Displays all historical Wordle answers reverse-chronologically with search and filter. Annotates entries with two badges:
- **reused** (amber) ‚Äî NYT recycled a past answer; shows original puzzle number
- **rare pick** (violet) ‚Äî NYT picked a word outside the curated ~2,300 answer pool, from the broader ~13K accepted-guess list

Includes an info modal explaining both phenomena. Stats bar shows live counts and rates.

### Analysis drawer

`src/lib/wordle/analysis.ts` ‚Äî pure functions (`analyzeGuess`, `letterCoverage`) that compute partition data and letter status on demand. No worker needed.

`src/components/AnalysisDrawer.tsx` ‚Äî side drawer (desktop 320px right panel, mobile bottom sheet). Triggered by "Why?" link on the solver recommendation. Shows stats summary, pattern outcomes with visual flow, and letter coverage grid. Also supports analyzing custom guesses with a comparison banner vs the solver's pick.

### Benchmark

CLI harness (`scripts/benchmark.ts`) plays all historical Wordle games using the solver and measures performance. Weight configs in `benchmark/configs/` define tunable parameters. Results are written to `benchmark/results/` (gitignored). Dashboard at `benchmark/dashboard.html` visualizes results.

Key files:
- `scripts/benchmark.ts` ‚Äî CLI entry point
- `scripts/benchmark-engine.ts` ‚Äî core game simulation loop
- `scripts/benchmark-stats.ts` ‚Äî summary statistics calculator
- `scripts/benchmark-types.ts` ‚Äî shared TypeScript types
- `benchmark/configs/` ‚Äî weight config JSON files
- `benchmark/results/` ‚Äî output result JSON files (gitignored)
- `benchmark/dashboard.html` ‚Äî standalone visualization dashboard
- `benchmark/CHANGELOG.md` ‚Äî tracks solver versions, configs, and historical results

Benchmark results include the git commit hash in the JSON and filename for traceability.
